Тема:

### 8. Складской учет

**Бизнес-контекст:** учет остатков, приходов, списаний и перемещений между складами.

**Роли и права:**

1. `WAREHOUSE_OPERATOR` оформляет движения товара.
2. `PROCUREMENT_MANAGER` управляет поставщиками и заявками на закупку.

**Обязательные сущности:**

1. `Warehouse`.
2. `Product`.
3. `StockBalance`.
4. `StockMovement`.
5. `Supplier`.
6. `PurchaseOrder`.
7. `PurchaseOrderItem`.

**Ключевые бизнес-правила:**

1. Остаток по товару на складе не может стать отрицательным.
2. Перемещение между складами должно менять остатки в обоих местах.
3. Приход возможен только на существующий склад и товар.
4. Списание возможно только при достаточном остатке.
5. Каждое движение фиксируется в истории с причиной операции.

**Обязательные пользовательские сценарии (MVP):**

1. Оператор оформляет приход от поставщика.
2. Оператор выполняет расход товара.
3. Оператор делает перемещение между складами.
4. Менеджер создает заявку на закупку при низком остатке.

**Минимальный API-контур:**

1. CRUD для `Warehouse`, `Product`, `Supplier`.
2. `POST /stock/inbound`.
3. `POST /stock/outbound`.
4. `POST /stock/transfer`.
5. `GET /stock/balances`.
6. CRUD для `PurchaseOrder`.

**Критерии приемки:**

1. Инвариант неотрицательных остатков не нарушается.
2. История движений согласована с фактическими остатками.
3. Операции перемещения атомарно отражаются на двух складах.
